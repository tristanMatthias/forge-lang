Annotations stable

@name and @name(args) — metadata annotations on declarations

Annotation system: @name and @name(args) on component fields, models, and routes

Annotations attach metadata to declarations using the `@name` or `@name(args)` syntax. They appear before functions, types, fields, and other declarations. For example, `@deprecated fn old() { }` marks a function as deprecated, and `@syntax("pattern")` configures a component syntax pattern.

Annotations are the primary extensibility mechanism for Forge's compiler and provider system. Rather than adding keywords for every new concept, Forge uses annotations to layer behavior onto existing syntax. This keeps the core language small while allowing rich, domain-specific features.

The compiler processes annotations during different compilation phases. Some annotations affect parsing (`@syntax`), some affect type checking (`@deprecated`), and some affect code generation. Provider templates can define custom annotations that control template expansion behavior.

This design is similar to Java annotations, Python decorators, and C# attributes. The key difference is that Forge annotations integrate with the template system, so providers can define new annotations without compiler changes.

Syntax

@name
@name(args)

/// Test: CRUD expose — @hidden fields excluded from model JSON, visible fields accessible
/// Tests the field exposure pattern that server `expose` syntax builds on.
///

use @std.model.{model}

model Account {
  id: int @primary @auto_increment
  username: string
  email: string
  password: string @hidden
  api_key: string @hidden
}

fn main() {
  Account.create({ username: "alice", email: "alice@test.com", password: "pass123", api_key: "key-abc" })
  Account.create({ username: "bob", email: "bob@test.com", password: "pass456", api_key: "key-xyz" })

  // List returns records without hidden fields
  let accounts = Account.list()
  let first_json = json.stringify(accounts)

  // Visible fields present
  println(string(first_json.contains("alice")))
  // Hidden fields absent
  println(string(first_json.contains("pass123")))

  // Individual get also hides fields
  let acc = Account.get_internal(1)
  let acc_json = json.stringify(acc)
  println(string(acc_json.contains("alice@test.com")))

  // Count confirms both records exist
  println(string(Account.count()))
}

crud_nested.fg

/// Test: Nested CRUD — create parent and children, update parent, cascade query
///

use @std.model.{model}

model Author {
  id: int @primary @auto_increment
  name: string
}

model Post {
  id: int @primary @auto_increment
  title: string
  author_id: int
}

fn main() {
  let author_id = Author.create({ name: "alice" })
  Post.create({ title: "First Post", author_id: author_id })
  Post.create({ title: "Second Post", author_id: author_id })

  // Query children by parent
  let posts = Post.where({ author_id: author_id })
  println(string(posts.length))
  println(posts[0].title)
  println(posts[1].title)

  // Update parent, children unaffected
  Author.update(author_id, { name: "Updated Author" })
  let author = Author.get(author_id)
  if author != null {
    println(author.name)
  }

  // Delete one child, verify count
  Post.delete(1)
  println(string(Post.count_where({ author_id: author_id })))

  // Delete parent, children still exist (no cascade)
  Author.delete(author_id)
  println(string(Author.count()))
}

/// Test: CRUD pagination — Model.paginate returns paginated results with metadata

  println(string(page1.current_page))
  println(string(page1.total_pages))

  // Page 2
  let page2_json = Post.paginate("{}", 2, 10, "id ASC")
  let page2: PageMeta = json.parse(page2_json)
  println(string(page2.current_page))
  println(string(page2.total_pages))
}
