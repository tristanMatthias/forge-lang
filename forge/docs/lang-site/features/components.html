template-driven component system with provider integration

Template-driven component system with provider architecture and lifecycle hooks

Components are Forge's template-driven extension system. A component defines a reusable, domain-specific abstraction backed by a provider. For example, a `model` component creates a data model with CRUD operations, a `server` component sets up an HTTP server, and a `queue` component provides message queue functionality.

Components are defined entirely through provider template files (`provider.fg`). The compiler has zero knowledge of any specific component; it simply expands templates by substituting placeholders. This means new component types can be added without modifying the compiler.

Using a component is as simple as writing a block: `model User { name: string, email: string }`. The compiler finds the matching template from the loaded providers, expands it with the user's schema and configuration, and produces plain Forge code that calls extern functions from the provider's native library.

This architecture separates concerns cleanly: providers implement behavior in native code, templates describe how to expose that behavior to Forge users, and the compiler handles the mechanical work of template expansion. Adding a new domain (database, message queue, GPU compute) requires only a new provider, never a compiler change.

Syntax

use @ns.name
name(args) { config }

Examples

model_crud.fg

/// # Model Component CRUD
///
/// The model component provides create, get, list, update, delete, count.
/// All component behavior is template-driven from provider.fg.
///

use @std.model.{model}

model Task {
  id: int @primary @auto_increment
  title: string
  done: bool @default(false)
}

fn main() {
  let id1 = Task.create({ title: "Write tests", done: false })
  let id2 = Task.create({ title: "Review code", done: true })
  let id3 = Task.create({ title: "Deploy", done: false })

  let task = Task.get(id1)
  if task != null {
    println(task.title)
  }

  println(string(Task.count()))

  Task.delete(id3)
  println(string(Task.count()))
}

model_validation_error.fg

/// # Model Validation Error
///
/// When create() returns 0, Model.last_error() returns structured
/// JSON describing which fields failed validation and why.
///

use @std.model.{model}

model User {
  id: int @primary @auto_increment
  name: string @min(1)
}

fn main() {
  let id = User.create({ name: "" })
  println(string(id))

  let err = User.last_error()
  // err is JSON like: {"__validation_error":true,"error":"validation failed","fields":[{"field":"name","rule":"min","message":"name must be at least 1 characters, got 0"}]}
  // Just check it contains "name"
  if err.contains("name") {
    println("name")
  }
}

service_hooks.fg

/// # Service with Hooks
///
/// Services wrap models with before/after event hooks.
///

use @std.model.{model, service}

model Item {
  id: int @primary @auto_increment
  name: string
  active: bool @default(true)
}

service ItemService for Item {
  before create(item) {
    println(`creating: ${item.name}`)
  }

  after create(item) {
    println(`created id: ${item.id}`)
  }
}

fn main() {
  let id = ItemService.create({ name: "Widget", active: true })
  println(string(ItemService.count()))
}
