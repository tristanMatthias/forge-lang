# Forge — Project Architecture

## Principle

Every language feature is a self-contained module. One directory = one feature. The directory contains everything: compiler code, tests, examples, and docs. A Rust macro handles registration so adding a feature is: create a module, write the code, done.

## The Macro

```rust
// compiler/core/feature_macro.rs

// Usage:
#[forge_feature(
    name = "Null Safety",
    status = "stable",                        // draft | wip | testing | stable
    depends = ["types_core", "pattern_matching"],
    enables = ["error_propagation"],
    tokens = [QuestionMark, QuestionDot, DoubleQuestion, Bang],
    ast_nodes = [SafeAccess, NullCoalesce, ForceUnwrap, NullableType],
    description = "Optional types with ?, ?., ??, and smart narrowing",
)]
pub mod null_safety;
```

The macro generates:

```rust
// Auto-generated by #[forge_feature]
impl Feature for null_safety {
    fn metadata() -> FeatureMetadata {
        FeatureMetadata {
            name: "Null Safety",
            id: "null_safety",
            status: FeatureStatus::Stable,
            depends: &["types_core", "pattern_matching"],
            enables: &["error_propagation"],
            description: "Optional types with ?, ?., ??, and smart narrowing",
        }
    }

    fn register_parser(p: &mut ParserRegistry) {
        null_safety::register_parser(p);
    }

    fn register_checker(c: &mut CheckerRegistry) {
        null_safety::register_checker(c);
    }

    fn register_codegen(g: &mut CodegenRegistry) {
        null_safety::register_codegen(g);
    }
}

// Auto-registers into the global feature list
inventory::submit!(FeatureEntry::new::<null_safety>());
```

The `inventory` crate collects all feature registrations at link time. No manual list to maintain.

## Inside a Feature Module

```rust
// compiler/features/null_safety/mod.rs

#[forge_feature(
    name = "Null Safety",
    status = "stable",
    depends = ["types_core", "pattern_matching"],
    enables = ["error_propagation"],
    tokens = [QuestionMark, QuestionDot, DoubleQuestion, Bang],
    ast_nodes = [SafeAccess, NullCoalesce, ForceUnwrap, NullableType],
    description = "Optional types with ?, ?., ??, and smart narrowing",
)]

// ── Parser ──

pub fn register_parser(p: &mut ParserRegistry) {
    p.postfix_type("?", parse_nullable_type);
    p.infix("?.", Precedence::Access, parse_safe_access);
    p.infix("??", Precedence::NullCoalesce, parse_null_coalesce);
    p.prefix("!", parse_force_unwrap);
}

fn parse_nullable_type(parser: &mut Parser, base: Type) -> Result<Type, Diagnostic> {
    Ok(Type::Nullable(Box::new(base)))
}

fn parse_safe_access(parser: &mut Parser, left: Expr) -> Result<Expr, Diagnostic> {
    let field = parser.expect_ident()?;
    Ok(Expr::SafeAccess { object: Box::new(left), field })
}

fn parse_null_coalesce(parser: &mut Parser, left: Expr) -> Result<Expr, Diagnostic> {
    let right = parser.parse_expression(Precedence::NullCoalesce)?;
    Ok(Expr::NullCoalesce { left: Box::new(left), right: Box::new(right) })
}

fn parse_force_unwrap(parser: &mut Parser, expr: Expr) -> Result<Expr, Diagnostic> {
    Ok(Expr::ForceUnwrap(Box::new(expr)))
}

// ── Type Checker ──

pub fn register_checker(c: &mut CheckerRegistry) {
    c.register(AstNode::SafeAccess, check_safe_access);
    c.register(AstNode::NullCoalesce, check_null_coalesce);
    c.register(AstNode::ForceUnwrap, check_force_unwrap);
}

fn check_safe_access(checker: &mut Checker, expr: &Expr) -> Result<Type, Diagnostic> {
    // left must be T?, result is the field type but nullable
    // ...
}

fn check_null_coalesce(checker: &mut Checker, expr: &Expr) -> Result<Type, Diagnostic> {
    // left must be T?, right must be T, result is T
    // ...
}

// ── Codegen ──

pub fn register_codegen(g: &mut CodegenRegistry) {
    g.register(AstNode::SafeAccess, emit_safe_access);
    g.register(AstNode::NullCoalesce, emit_null_coalesce);
    g.register(AstNode::ForceUnwrap, emit_force_unwrap);
}

fn emit_safe_access(ctx: &mut CodegenCtx, expr: &Expr) -> Result<LLVMValue, Diagnostic> {
    // if left != null { left.field } else { null }
    // ...
}

fn emit_null_coalesce(ctx: &mut CodegenCtx, expr: &Expr) -> Result<LLVMValue, Diagnostic> {
    // if left != null { left } else { right }
    // ...
}

#[cfg(test)]
mod tests;
```

## Examples as Tests and Docs

```forge
// compiler/features/null_safety/examples/safe_access.fg

/// # Safe Access
///
/// Use `?.` to safely access properties on nullable values.
/// If the value is null, the entire chain evaluates to null
/// instead of crashing.
///
/// expect: anonymous

fn main() {
  let user: {name: string}? = null
  let name = user?.name ?? "anonymous"
  println(name)
}
```

```forge
// compiler/features/null_safety/examples/narrowing.fg

/// # Smart Narrowing
///
/// After an `if` null check, the type is automatically narrowed.
/// No need to unwrap manually inside the branch.
///
/// expect: alice

fn main() {
  let name: string? = "alice"
  if name != null {
    // name is string here, not string?
    println(name.upper())
  }
}
```

The `/// expect:` line makes it a test. The `///` block is extracted as documentation. The code is a runnable example. One file, three uses.

Running them:

```bash
# Run all tests for a feature
cargo test --features null_safety

# Run example files as integration tests
forge test features/null_safety/examples/

# Generate docs from examples
forge docs features/null_safety/
```

## How `forge features` Works

It's not a separate database. It reads from the compiled binary's feature registry:

```rust
// compiler/core/registry.rs

pub struct FeatureRegistry {
    features: Vec<FeatureMetadata>,
}

impl FeatureRegistry {
    pub fn all() -> &[FeatureMetadata] {
        // inventory crate collects all #[forge_feature] entries at link time
        inventory::iter::<FeatureEntry>()
            .map(|e| e.metadata())
            .collect()
    }

    pub fn get(id: &str) -> Option<&FeatureMetadata> { ... }
    pub fn graph() -> FeatureGraph { ... }
    pub fn by_status(status: FeatureStatus) -> Vec<&FeatureMetadata> { ... }
}
```

Test counts are computed by running `forge test` on the examples directory. Status is from the macro. Dependencies are from the macro. Nothing to maintain — it's all derived from code.

## Adding a New Feature — Step by Step

```bash
# 1. Create the directory
mkdir -p compiler/features/my_feature/examples

# 2. Write the module with the macro
cat > compiler/features/my_feature/mod.rs << 'EOF'
#[forge_feature(
    name = "My Feature",
    status = "draft",
    depends = [],
    description = "Does something cool",
)]

pub fn register_parser(p: &mut ParserRegistry) { }
pub fn register_checker(c: &mut CheckerRegistry) { }
pub fn register_codegen(g: &mut CodegenRegistry) { }

#[cfg(test)]
mod tests;
EOF

# 3. Write an example (which is also a test and docs)
cat > compiler/features/my_feature/examples/basic.fg << 'EOF'
/// # Basic usage
///
/// Shows the simplest use of my feature.
///
/// expect: hello

fn main() {
  println("hello")
}
EOF

# 4. Register in features/mod.rs
echo "pub mod my_feature;" >> compiler/features/mod.rs

# 5. Verify it shows up
forge features | grep my_feature
# my_feature               draft       0/1  ○   -
```

That's it. Five steps. No TOML to write. No registration code. No updating a master list.

## Migration Plan

The current codebase has everything in big files. Migration is incremental:

1. Create the `features/` directory and `core/` directory
2. Set up the `#[forge_feature]` macro and `inventory` dependency
3. Extract ONE feature (start with the simplest — `immutability` or `pipe_operator`)
4. Verify all tests pass
5. Extract the next feature
6. Repeat until the old monolithic files are empty and deleted

Each extraction is a PR. Each PR is green. No big bang rewrite.

### Order of extraction (simplest first):

1. `immutability` — let/mut/const
2. `pipe_operator` — |>
3. `string_templates` — backtick interpolation
4. `ranges` — 0..10, 0..=10
5. `for_loops`
6. `closures`
7. `it_parameter`
8. `null_safety`
9. `error_propagation`
10. `pattern_matching`
11. `traits`
12. `generics`
13. `extern_ffi`
14. `c_abi_trampolines`
15. `json_builtins`
16. `components`
17. `channels`
18. `spawn`
19. `select_syntax`
20. `shell_shorthand`

## CLAUDE.md Update

```markdown
## Project Structure

Features are self-contained modules in `compiler/features/`.
Each feature has: mod.rs (parser + checker + codegen), tests.rs, and examples/.

### Adding a feature
1. Create directory in compiler/features/
2. Add #[forge_feature] macro with metadata
3. Implement register_parser, register_checker, register_codegen
4. Add examples with /// expect: comments (these are tests AND docs)
5. Add pub mod line to features/mod.rs

### Finding code for a feature
Everything for a feature lives in its directory. Don't grep — go to the directory.
`forge features` shows all features with status and test counts.
`forge features <name>` shows details for one feature.

### Rules
- Never put feature-specific code in core/
- Core is infrastructure only: lexer, AST types, type system, codegen context
- Every example file must have a /// expect: comment
- Feature status must be accurate: draft (no tests), wip (some pass), testing (most pass), stable (all pass)
```
